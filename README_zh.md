# devclean

> 一个安全、可解释的开发目录分析器。
>
> **devclean 不会删除任何内容。**

---

## 什么是 devclean？

`devclean` 扫描开发目录并**解释哪些文件夹可能是安全的，可以删除——以及原因**。

它是为以下类型的开发者设计的：

* 使用**多种语言或框架**工作的开发者
* 维护**单体仓库（monorepos）或长期项目**的开发者
* 有一些怀疑是构建产物，但**不敢轻易删除**的目录

与传统的清理工具不同，devclean 专注于**分析和建立信任**，而不是自动化操作。

---

## 它解决了什么问题？

大多数开发者最终都会面对这样的目录：

```text
./target/
./node_modules/
./build/
./dist/
./.cache/
```

你*认为*它们是可以安全删除的——但你并不是 100% 确定。

**如果 devclean 或多或少让你有些犹豫，请提出一个 issue。**

目前的选择都不尽如人意：

* `rm -rf` —— 速度快但非常危险
* `git clean -xfd` —— 甚至更危险
* 特定语言的工具 —— 只能在单一生态系统中工作

`devclean` 的存在就是为了回答一个问题：

> **“我可以删除这个目录吗？”**

同样重要的是：

> **“你为什么这么认为？”**

---

## devclean 做什么（以及不做什么）

### ✅ 它做什么

* 检测常见的项目类型（Rust, Node.js, Python, Java, …）
* 识别**众所周知的构建或依赖目录**
* 使用清晰的证据解释其推理过程
* 为每个建议分配一个置信度（Confidence level）
* 仅在**试运行模式（dry-run mode）**下运行

### ❌ 它不做什么

* ❌ 它**不会**删除文件或目录
* ❌ 它**不承诺** 100% 的正确性
* ❌ 它**不会**扫描你的整个系统
* ❌ 它**不会**尝试使用机器学习或启发式算法来“耍小聪明”

如果你想要一个自动清理工具，这可能**不**适合你。

---

## 示例用法

```bash
$ devclean scan .
```

示例输出：

```text
发现 4 个值得审查的目录：

✔ target/          Rust Cargo 构建目录
  证据：在项目根目录找到 Cargo.toml
  置信度：高

✔ node_modules/    Node.js 依赖项
  证据：找到 package.json
  置信度：高

⚠ .cache/          混合使用的缓存目录
  证据：名称匹配常见的缓存模式
  置信度：低

✖ src/             源代码目录
  原因：受保护的路径
```

决定权在你——devclean 只提供上下文信息。

## 支持的生态系统（初始版本）

* Rust (Cargo)
* Node.js (npm / yarn / pnpm)
* Python (venv / 基于 pyproject 的工作流)
* Java (Maven / Gradle)

支持策略是有意保持保守和基于规则的。

---

## 设计理念

1. **安全优于便利**
2. **可解释性优于聪明**
3. **明确的不确定性优于虚假的自信**

一个你理解的建议，胜过一个让你后悔的操作。

## 非目标（Non-goals）

为了保持项目的专注，devclean 明确避免：

* 一键清理
* 系统级缓存清除
* 特定 IDE 或编辑器的清理
* 没有解释的“魔法”检测
